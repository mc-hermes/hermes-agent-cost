gbrain MVP — Email-First Architecture

The demo that sells itself: forward an email, watch a brain grow. Zero OAuth. Zero Drive. Three moving parts.

1. The Demo Moment

The scene that sells the product

You pre-provision a brain for a prospect. You give them one thing: an email address.

zaim+test@brains.malohacoast.com

Zaim opens Gmail. Forwards a meeting transcript. Hits send.

You both watch brains.malohacoast.com/zaim-test — the dashboard is already open. Within 30 seconds, a new node appears in his knowledge graph. The entity card populates. Links form. The brain score ticks up.

"That was my email. That's my brain. I didn't install anything."

That's the product. The rest is pricing.

4. The MVP Components (Build These First)

#ComponentStackLinesWhat it does
1 Provisioning Script Bash ~60 provision-brain.sh zaim-test → creates container, installs Bun + gbrain, spins up SMTP receiver + API server + dashboard, configures DNS. One command per demo brain.
2 SMTP Receiver Python (aiosmtpd) ~80 Listens on :25 inside each container. Accepts mail to *@brains.mlhc. Parses recipient (extracts brain name from zaim+test@...), writes body+attachments to ~/workspace/YYYY-MM-DD-slug.md. No auth — SMTP is wide open (rate-limited by container).
3 Filesystem Watcher Python (watchdog) ~50 Watches ~/workspace/ for new .md files. On new file, triggers Hermes pipeline. Debounces 5s to batch rapid emails. Runs as a daemon inside the container.
4 Hermes Pipeline Hermes Agent (existing) ~0 Already built. gbrain put → extract all → embed --all → doctor → export JSON. Wired to the FS watcher trigger.
5 API Server Fastify (Node) ~120 Two endpoints: POST /api/ask (query brain + LLM synthesis) and POST /api/ingest (dashboard upload, optional for MVP). Serves static dashboard files. Handles CORS for dev.
6 Dashboard (polling) Static HTML (existing) ~20 added The existing gbrain dashboard, with one addition: a 15-second polling loop that re-fetches gbrain-data.json. When brain changes, dashboard updates in-place — no page reload. Graph springs to life. New node appears. Pulse dot at top shows "live" status.
7 Reverse Proxy Caddy ~10 Wildcard TLS for *.brains.malohacoast.com. Routes zaim-test.brains.mlhc → container IP :3001. Handles Let's Encrypt automatically.
8 DNS Cloudflare ~2 records MX record: brains.mlhc → VPS IP. Wildcard A: *.brains.mlhc → VPS IP. Done.

Total new code: ~340 lines (60 bash + 80 SMTP + 50 watcher + 120 API + 20 dashboard + 10 Caddy). Most of the heavy lifting is existing gbrain + Hermes. Buildable in a focused weekend.

5. The Demo Flow — Step by Step

1

Provision the brain (before the meeting)

provision-brain.sh zaim-test — 30 seconds. Creates container, installs everything, dashboard live at zaim-test.brains.malohacoast.com. Email zaim+test@brains.malohacoast.com is active. Open the dashboard on your screen.

2

Zaim forwards an email

"Hey Zaim, forward me that meeting recap from last week. Just send it to zaim+test@brains.mlhcoast.com." He opens Gmail, hits Forward, types the address, hits Send. That's it. Zero install.

3

The email lands (~3 seconds)

SMTP receiver in the container accepts the mail. Parses subject + body. Writes ~/workspace/2026-06-08-meeting-with-nabil.md. The FS watcher fires an event.

4

Hermes processes (~15–45 seconds)

Pipeline triggers: gbrain put creates the page, extract all finds links and entities, embed --all generates vectors (OpenAI call), doctor scores the brain, enrichment script exports gbrain-data.json. Total: 15-45 seconds depending on OpenAI latency.

5

Dashboard updates (~15 seconds)

Dashboard's polling loop fetches the new gbrain-data.json. The knowledge graph re-renders — a new node appears for "Nabil Jalil." The entity card populates with details. The page count ticks from 3 to 4. The brain score updates. The "Last updated" timestamp changes to "just now."

6

Zaim asks his brain a question

"Now Zaim, ask it a question. Type 'what did I discuss with Nabil?'" He types it into the dashboard. POST /api/ask → gbrain query → OpenAI GPT-4o synthesis → answer with source citations, including the page he just created by forwarding an email 60 seconds ago.

Total time from Zaim pressing Send to seeing his knowledge graph update: under 60 seconds. That's the demo.

☁️ Google DriveAdd as premium feature for clients who want auto-sync. Not needed for email-first demo.
📱 WhatsApp/TelegramSame SMTP pattern — WhatsApp → email gateway → brain. Add when a client asks.
⚡ WebSocket/SSE15-second polling is fine for MVP. Replace with SSE for push updates in post-MVP polish.

8. The Email Addressing Scheme

The SMTP receiver parses the recipient address to route to the right brain:

zaim+test@brains.malohacoast.com
│    │
│    └─ Brain name (maps to container)
└────── Client identifier (for routing / future multi-brain per client)

For MVP, keep it flat: {anything}@brains.mlhcoast.com → the + suffix is the brain name. The container's SMTP receiver accepts everything and routes based on the +name part.

A single catch-all MX record on Cloudflare routes *@brains.mlhcoast.com to your VPS. The SMTP receiver on the host (not per-container) parses the recipient, looks up which container handles that brain, and forwards the mail. Alternatively: a single shared SMTP receiver that writes to a shared workspace and a routing table.

Simplest MVP approach: one SMTP receiver per container, each listening on a different port. The host-level reverse proxy (or iptables) routes zaim+test@... → container on port 2501, fatima+coach@... → container on port 2502. The MX record points everything to the VPS. Done.

10. What This Architecture Is NOT

Not a SaaS platform. It's a set of provisioned demo brains. No billing. No auth. No multi-tenancy management. No admin UX. That's the point — the MVP proves the value proposition (email → brain → insight) with the absolute minimum surface area. SaaS features are layered on after someone says "I'll pay for this."

Not scalable to 100 clients. But it scales to 10 demos easily. By the time you have 10 demos running, you'll know exactly what the paying version needs.

Not the final architecture. This is step 1. The full single-tenant architecture from the previous writeup is step 20. Don't build step 20 until step 1 proves people want it.
