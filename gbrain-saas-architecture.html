gbrain SaaS Architecture

Deploying a "second brain" product for non-technical clients — single-tenant managed hosting with BYO-keys. June 2026.

1. What Is the Product?

gbrain is a knowledge-graph engine — a Bun CLI with an embedded PGLite vector store. It ingests raw material (meeting transcripts, articles, voice notes, PDFs), extracts entities and wiki links, generates OpenAI embeddings, and builds a queryable knowledge graph.

Hermes Agent is the AI operator layer that runs the processing pipeline: ingest → extract entities → embed → enrich → synthesize Q&A. The agent calls gbrain put, gbrain embed --all, gbrain extract all, and gbrain doctor on a schedule.

The dashboard is a static HTML frontend — a single file that fetches an exported gbrain-data.json and renders an interactive knowledge graph with search, entity cards, and health panels. No backend. No framework. Deployable anywhere.

The goal: productize this as a monthly subscription for solo coaches and consultants who never touch a terminal.

2. The Core Constraint

PGLite Is an Embedded Database — Not a Server

PGLite is a single-file WASM PostgreSQL that runs inside the Bun process. There is no network port. No client/server protocol. No remote connection. The gbrain CLI must run on the same machine where the brain.pglite file lives. This single fact determines everything about the deployment architecture.

This means the core question is: who runs the Bun process, and on whose machine does the PGLite file sit?

4. Is This a SaaS?

Yes — but a specific kind: single-tenant managed platform with BYO-keys.

It is not multi-tenant SaaS (shared database, shared runtime) because PGLite does not support multi-tenancy natively. Each client gets their own gbrain process + PGLite file. You cannot pool them into one database without replacing PGLite with a full PostgreSQL server — which would defeat the local-first architecture of gbrain.

It is not pure self-hosted because you run the infrastructure.

It is not pure desktop software because the runtime lives on your servers.

The closest analogues:

  • Ghost(Pro) — managed single-tenant Ghost blogs. Each customer gets their own instance. BYO domain.
  • WordPress.com (Business tier) — managed WP instances with plugin isolation.
  • Supabase (dedicated) — managed Postgres per project.

The pricing model works because:

  • Client pays you a monthly subscription ($29–99/mo depending on page tiers, query volume, features)
  • Client brings their own OpenAI API key — you never see an OpenAI bill
  • Your margin = subscription minus infrastructure (~$2–5/mo/client on a modest VPS)

5. Recommended Architecture

Managed Single-Tenant LXC Containers

Every client gets a dedicated LXC container on your VPS. Inside that container lives: Bun, gbrain CLI, PGLite, the Hermes agent, a Google Drive sync daemon, and a lightweight Query API for LLM-powered Q&A.

9. Trade-offs of the Recommended Architecture

Privacy Compromise

Raw files transit through your VPS during processing. A privacy-obsessed client might object. Mitigated by: clear data handling policy in ToS, encryption at rest, full data purge on cancellation, BYO-keys. If a client absolutely requires "my data never touches a third-party server," offer a local app tier (Option A) as an enterprise plan.

PGLite Is Not Concurrent

Only ONE gbrain process can write to brain.pglite at a time. If Hermes is mid-build and the client hits /api/ask, the query could fail or get stale results. Solution: job queue. Builds are queued. Queries read from a read-only copy or wait. Flagged for implementation.

Container Overhead

Each LXC is ~200–400 MB RAM (Bun + gbrain + daemons). At 50 clients: 10–20 GB RAM on your VPS. Manageable on a $40–80/mo VPS. At 200+ clients, you'll need orchestration (Kubernetes or multiple VPSs).

Build Time

First build for a client with 50 files might take 5–15 minutes (embeddings are the bottleneck — OpenAI's API is fast but rate-limited). Client can't use the brain until the build finishes. Need a progress indicator in the dashboard.

OpenAI Key Exposure

The key lives in the container env. If your VPS is compromised, keys could be exfiltrated. Mitigation: container isolation, no logging of env vars, read-only mount from a secrets manager, and alerting on anomalous OpenAI spend.

Dashboard Is Stale Between Builds

The dashboard fetches a JSON snapshot. It doesn't auto-refresh. Solution: cron-based rebuild every 6 hours + manual "Refresh Brain" button. For real-time updates, add a WebSocket — but that's overkill for solo coaches.

10. Failure Modes

FailureImpactMitigation
Client's OpenAI key expires Embeddings fail, queries fail. Brain goes stale. Health check monitors key validity. Email alert to client. Graceful degradation: search still works (keyword only), no LLM synthesis.
Google Drive OAuth token expires Sync daemon can't pull new files. Token refresh logic in rclone. If refresh fails, email client to re-auth.
PGLite file corruption Brain is dead. All data lost. Nightly PGLite snapshots to backup store. Restore from last good snapshot. gbrain doctor detects corruption early.
Container runs out of disk Builds fail, queries fail. Disk quota monitoring. Auto-alert at 80% usage. Auto-prune old workspace files.
OpenAI API is down Embeddings and LLM queries fail. Queue and retry with exponential backoff. Client sees "OpenAI is temporarily unavailable — your brain search still works."
VPS goes down All clients' brains unavailable. Single point of failure. Mitigate with standby VPS + replicated containers, or accept 99.5% uptime as acceptable for a solo-coach tool.

11. gbrain Internals That Constrain the Design

Confirmed (from live gbrain instance)

  • PGLite is a single file (brain.pglite) opened by the Bun process. No network port. No client/server protocol. This is the single biggest constraint — it means gbrain MUST run locally to the PGLite file.
  • gbrain uses OpenAI for embeddings (text-embedding-3-small). API call per page during gbrain embed --all. Client's key used server-side. No local embedding model needed.
  • The CLI is synchronous. gbrain put, gbrain embed, gbrain extract are blocking commands. No streaming. A build is a sequential pipeline.
  • Export is file-based: gbrain export --dir /tmp/brain-export dumps markdown files. The enrichment script (Python, stdlib only) builds the dashboard JSON. This is a batch operation.
  • The dashboard is pure static HTML + JSON fetch. No backend needed for browsing. This is a huge advantage — it means the dashboard can be hosted on a $0 CDN and only the Query API needs compute.

Guesses (needs verification)

  • Does gbrain have built-in auth or multi-user support? I'm assuming none. The container is the isolation boundary. You'd need to build auth yourself.
  • Does gbrain query return structured data? I'm assuming JSON or parseable text. If it only prints human-readable output, you'll need a wrapper script to parse it for the Query API.
  • Memory/CPU profile at scale — I've seen it handle ~30 pages fine. At 500+ pages, the Bun process might need tuning. Benchmark before production.
  • Does PGLite support WAL or read replicas? If it does, you could have a read replica for queries while the main process builds. If not (likely — it's embedded), use the job queue approach.

12. The Scale Path

Phase 1 (1–10 clients):
  1 VPS ($40/mo) → 10 LXC containers → manual provisioning

Phase 2 (10–50 clients):
  2–3 VPSs → Ansible/Terraform provisioning → admin panel for health

Phase 3 (50–200 clients):
  Kubernetes cluster → per-client namespaces → automated billing
  → offer "bring your own cloud" for enterprise clients

Phase 4 (200+ clients):
  Multi-region → dedicated embedding models (no OpenAI dependency)
  → tiered storage (hot PGLite + cold S3 archive)
