1 Problem Statement & Current-State Analysis

1.1 The Knowledge Ingestion Gap

Maloha Coast operates a knowledge graph (gbrain) that stores structured intelligence — people, companies, meetings, concepts, and their interconnections — in a PGLite-backed knowledge base. The current brain holds 37 pages, 30 graph edges, 10 people, and 10 companies with a composite brain score of 28/37.

However, valuable knowledge arrives outside the brain — in newsletters, forwarded articles, client emails, event announcements, and research digests. The user (sayf) forwards these from sayfcodes@gmail.com to info@malohacoast.com. Before this pipeline, each forward required a manual cycle: read the email, extract entities, decide where it fits in the brain, write a brain page, link it to related pages, embed it, and deploy the dashboard update. This manual cycle took 5-15 minutes per email and was frequently deferred or skipped entirely.

Current-State Pain Points

  • Knowledge loss: Forwards sitting unprocessed in the inbox represent lost institutional memory.
  • Inconsistent structure: Manual brain entries vary in format, tagging, and link density.
  • High cognitive overhead: Reading, extracting, writing, linking, and deploying for every forward is a context switch.
  • Dashboard staleness: Without automated ingestion, the dashboard reflects periodic bursts rather than continuous knowledge flow.

2 Solution Overview — The Email-Forwarding Paradigm

2.1 Core Idea

The pipeline treats email as an ingestion API. The user forwards any piece of knowledge to info@malohacoast.com — a newsletter, a meeting summary, an article link, a LinkedIn profile, or raw notes. The system polls the inbox every 2 minutes, processes qualifying emails through a multi-phase pipeline, creates structured brain pages with entity extraction and vector embeddings, and replies to confirm ingestion. Total latency: 90-120 seconds from forward arrival to brain page live.

Paradigm shift: The user's only action is forwarding an email. Everything else — reading, extracting, classifying, embedding, linking, deploying, and replying — is automated. Zero-touch ingestion.

2.2 User Flow

  1. Receive — User gets an email they want in the brain (in personal inbox sayfcodes@gmail.com).
  2. Forward — User forwards it to info@malohacoast.com.
  3. Wait ~2 min — Pipeline polls, processes, ingests, and embeds automatically.
  4. Receive confirmation — User gets a reply email with the brain page slug, extracted entities, and a link to the dashboard.
  5. Verify (optional) — User can visit the brain page or dashboard to verify.

Design Principles

  • Email as universal input. Everything can be forwarded — newsletters, meeting notes, web clippings, LinkedIn bios.
  • Git as source of truth. Brain pages are Markdown in a git repo. Every ingestion is a commit. Undo is git revert.
  • Idempotent by Gmail message ID. Processed emails are labeled; re-polling the same email is a no-op.
  • Fail gracefully, never silently. Errors produce reply emails with diagnostics, not silent drops.
  • Zero ongoing cost. GitHub Pages hosting is free. Gmail API is free-tier. Embedding cost is ~$0.0001 per page.

3 Architecture & Pipeline Phases

3.1 High-Level Architecture

The pipeline is a Hermes cron-triggered batch processor with seven serial phases. Each phase either passes data forward or short-circuits with a reply. There is no persistent queue — the Gmail inbox IS the queue, and Hermes cron is the scheduler.

4 Component Specification

4.1 Hermes Cron Scheduler

Cron Job ID776c7bf32c77

Cron rationale: 2 minutes balances freshness (user waits at most 2 min for confirmation) against API rate limits. Gmail API free tier allows 1 billion quota units/day; this pipeline consumes ~720 searches/day (~72,000 units), well below the limit.

4.3 gbrain CLI

gbrain is the knowledge graph engine installed at ~/gbrain/ with the brain database at ~/.gbrain/brain.pglite. The pipeline invokes these CLI commands:

CommandPhasePurpose

gbrain doctor --jsonEnrichValidates brain health post-ingestion

4.4 Entity Extraction (LLM)

The most computationally intensive phase. The email body is passed to DeepSeek v4 Flash with a structured extraction prompt. The LLM returns a JSON object with:

  • Page type classification (concept, person, company, meeting)
  • Title suggestion based on email content
  • People mentioned — names, emails, roles
  • Companies mentioned — names, websites, industries
  • Topics — key concepts for tagging
  • Timeline entries — dates and events extracted from the body
  • Wikilinks — suggested links to existing brain pages

This phase accounts for ~40s of the pipeline latency. It is the only phase that uses an LLM. Future optimization: batch multiple emails into a single LLM call.

4.5 Enrichment Script

The Python enrichment script lives at: 

~/.hermes/skills/software-development/gbrain-dashboard/references/enrichment-script.py

It performs post-processing after the core ingestion:

  • Validates that all wikilink targets exist in the brain (flags orphans)
  • Computes link graph statistics (outbound count, inbound count, orphan count)
  • Generates the enriched gbrain-data.json for the dashboard
  • Runs gbrain doctor and embeds health check results
  • Updates the brain score computation

4.6 GitHub Pages Deploy

The final phase commits all changes and pushes to mc-hermes/hermes-agent-cost: 

cd ~/.gbrain
git add -A
git commit -m "ingest: <slug> from email <message-id>"
cd /path/to/hermes-agent-cost
python3 enrichment-script.py   # regenerates gbrain-data.json
cp /tmp/gbrain-data.json .
git add gbrain-data.json
git commit -m "dashboard refresh $(date -Iminutes)"
git push origin main

GitHub Pages rebuilds automatically on push. The dashboard is live within 60 seconds of git push.

6 Whitelist & Security Model

6.1 Sender Whitelist

The pipeline only processes emails from explicitly whitelisted senders. All other emails to info@malohacoast.com are ignored by the poll query.

EmailIdentityRoleStatus
sayfcodes@gmail.comsayfDirector, Maloha CoastACTIVE
Current state: Single-whitelist (sayfcodes@gmail.com). The whitelist is stored in the Hermes cron job configuration and can be extended to trusted team members or automation accounts (e.g., zapier@malohacoast.com for newsletter auto-forwarding).

6.2 Gmail Query Filter

The poll query uses Gmail's search syntax to pre-filter at the API level:

q=in:inbox is:unread from:sayfcodes@gmail.com -label:gbrain-processed

This ensures:

  • Sender-gated: Only whitelisted senders pass the from: filter.
  • Read-gated: Only unread emails are candidates (prevents re-processing).
  • Label-gated: The -label:gbrain-processed exclusion double-locks idempotency.

6.3 Post-Ingestion Labeling

After successful ingestion, the pipeline applies the Gmail label gbrain-processed to the message. This label is created once at pipeline setup via users.labels.create. The label serves as the idempotency token — the poll query excludes labeled messages.

6.4 Security Boundaries

8 Failure Modes & Recovery

8.1 Failure Catalog

Git push conflict 6 (Push) Non-fast-forward rejection Cron run logs error; dashboard stale Manual: git pull --rebase && git push
Brain DB corruption Any gbrain CLI errors on read/write gbrain doctor returns red checks Restore from git history (Markdown is source of truth; PGLite is derived)
Partial ingestion (crash mid-pipeline) 3-6 Page created but unlinked, unembedded, or un-pushed gbrain doctor flags orphan/partial pages Automatic: Gmail label NOT applied (UNREAD persists); re-processed next cron
Spam / unwanted forward 1 (Poll) Non-whitelisted sender Gmail query filter excludes it No action needed — email stays in inbox, never processed

8.2 Idempotency Guarantee

The pipeline achieves idempotency through the Gmail processed label. The label is applied ONLY after all phases succeed (ingest → entity → embed → enrich → push). If any phase fails, the label is not applied, the email remains UNREAD, and the next cron tick re-attempts the full pipeline.

Critical Safety Property

The Gmail label is the last operation in the pipeline. This means a crash at any point before labeling leaves the email UNREAD — it will be retried. A crash after labeling but before reply means the email is processed (no double-ingestion) but no confirmation sent. This is the safer failure direction (missing confirmations are visible to the user; duplicate brain pages are not).

9.3 Next Steps (Priority-Ordered)

P0 — Immediate (This Week)

  • Add monitoring cron: A separate daily cron that runs gbrain doctor and alerts on red checks.
  • Improve reply email: Include the brain page preview (first 200 chars) and direct link to the brain page slug.
  • Whitelist expansion: Add any additional trusted senders (team members, automation accounts).

P1 — Short-Term (Next 2 Weeks)

  • Reply-to-pipeline: Allow the user to reply to the confirmation email with corrections ("wrong type, should be meeting") — pipeline reprocesses.
  • Entity deduplication: Before creating a new person/company page, check for existing by email/domain fuzzy match.
  • Newsletter auto-forward: Set up Zapier/Make rule to auto-forward specific newsletters (AI Automation Society, System Design One) to info@malohacoast.com.

P2 — Medium-Term (Next Month)

  • Batch LLM extraction: Queue up to 5 emails and extract entities in a single LLM call — reduces per-email latency from 40s to ~10s.
  • Pipeline metrics dashboard: Track ingestion volume, LLM cost, error rate, and brain growth over time.
  • Multi-whitelist with roles: Different processing rules per whitelisted sender (e.g., client emails → meeting type; newsletters → concept type).

P3 — Long-Term (Quarter+)

  • Multi-brain support: Route emails to different gbrain instances per client.
  • Split JSON architecture: For brains >1,000 pages, split gbrain-data.json into summary + per-page files.
  • Webhook ingestion: Alternative to email forwarding — accept POSTs from Zapier, Make, n8n directly.
